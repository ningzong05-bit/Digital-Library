# 链路设计测试表.docx

## Paragraphs

1. 一、读者端核心业务链路
2. 1. 游客浏览与登录引导链路
3. 游客进入系统
→ 查看公告 / 检索公开馆藏
→ 查看图书可借状态和大致排队情况
→ 尝试借书 / 排队 / 查看个人权益 / 填写地址
→ 系统引导登录或注册
4. 对应功能：游客浏览公告、检索图书、查看基础馆藏信息。
6. 2. 用户登录与权限链路
7. 用户输入账号、密码、验证码
→ 系统校验身份
→ 登录成功生成访问令牌
→ 根据角色展示菜单
8. 角色包括：游客、读者、图书管理员、系统管理员。云端图书馆借阅系统V2.0.docx
10. 3. 读者资料与地址维护链路
11. 读者登录
→ 进入个人中心
→ 维护手机号、邮箱、收货地址、所在城市
→ 系统校验地址是否在图书馆所在市内
→ 保存有效地址
12. 这个链路会影响后面的“图书馆邮寄借书”，因为文档规定邮寄地址必须在同城范围内。
14. 4. 图书检索与详情查看链路
15. 游客 / 读者进入图书检索页
→ 按书名、作者、出版社、ISBN、分类、馆藏位置检索
→ 查看图书列表
→ 进入图书详情
→ 查看封面、简介、总馆藏、可借数量、排队人数、预计最快可借时间
16. 这是所有借阅行为的入口。
18. 5. 线上借书链路：到馆自取
19. 读者选择图书
→ 点击线上借书
→ 选择到馆自取
→ 系统校验年费、押金、借阅资格、可借库存、借阅额度
→ 校验通过后自动生成待自取记录
→ 管理员确认到馆自取交付
→ 系统记录自取确认日期
→ 开始计算 1 个月外借期限
20. 这个链路没有管理员审批，管理员只负责实际交付确认。
22. 6. 线上借书链路：图书馆邮寄
23. 读者选择图书
→ 点击线上借书
→ 选择图书馆邮寄
→ 填写市内收货地址
→ 系统校验年费、押金、借阅资格、可借库存、市内地址、年度寄送次数
→ 校验通过后生成借阅订单
→ 管理员登记馆方寄出日期
→ 配送费用按现结处理
→ 系统记录配送状态
→ 从馆方寄出日开始计算 1 个月外借期限
24. 这是文档里比较核心的“云端借阅 + 同城配送”链路。
26. 7. 无库存排队预约链路
27. 读者查看图书详情
→ 发现无可借库存
→ 点击加入排队
→ 系统校验年费、押金、借阅资格、排队数量
→ 生成排队顺序
→ 根据前方人数和每人最多外借 1 个月计算预计最快可借时间
→ 图书归还后库存恢复
→ 系统自动校验队首读者资格
→ 正常则自动生成借阅订单
28. 异常情况：
29. 队首读者资格失效 / 押金不足 / 超过额度
→ 系统暂停或跳过该排队记录
→ 管理员处理异常
→ 系统继续匹配下一位读者
31. 8. 借阅与配送记录查看链路
32. 读者登录
→ 进入我的借阅
→ 查看当前借阅 / 历史借阅
→ 查看取书方式、配送状态、寄出日期、自取日期、最晚还书日期、还书状态
33. 这个链路主要用于读者跟踪自己的借阅和物流状态。
35. 9. 读者发起还书链路：到馆还书
36. 读者进入我的借阅
→ 选择待归还图书
→ 选择到馆还书
→ 系统提示最晚到馆归还日期和费用规则
→ 提交还书申请
→ 借阅记录进入待验收状态
→ 管理员到馆验收
→ 系统判断是否逾期 / 损坏
→ 正常则更新为已归还并恢复库存
38. 10. 读者发起还书链路：邮寄还书
39. 读者进入我的借阅
→ 选择待归还图书
→ 选择邮寄到图书馆
→ 系统提示最晚寄出还书日期和到付规则
→ 读者提交还书申请
→ 借阅记录进入待收件状态
→ 管理员收到邮寄图书
→ 登记用户寄出还书日期和图书状态
→ 系统判断是否逾期 / 损坏
→ 正常则恢复库存
40. 文档规则里明确：读者邮寄还书以“用户寄出日”作为期限截止判断日期，不是图书馆实际收到日。
42. 二、图书管理员业务链路
43. 11. 图书入库链路
44. 图书管理员登录
→ 进入图书管理
→ 新增图书
→ 填写书名、ISBN、作者、出版社、分类、数量、是否支持配送
→ 系统保存图书信息
→ 馆藏数量增加
45. 如果 ISBN 重复，系统提示增加馆藏数量，而不是重复建档。
47. 12. 图书编辑链路
48. 管理员进入图书管理
→ 查询目标图书
→ 编辑基础信息、库存、馆藏位置、配送属性
→ 保存
→ 系统更新图书信息
49. 如果图书存在未归还借阅记录，删除操作受限制。
51. 13. 图书下架链路
52. 管理员选择目标图书
→ 点击下架
→ 填写下架原因
→ 系统检查是否存在未归还记录或有效排队
→ 无异常则更新为已下架 / 不可借
54. 14. 自取与邮寄发货确认链路
55. 管理员进入待自取 / 待寄出列表
→ 查看读者、图书、取书方式
→ 到馆自取：确认交付
→ 图书馆邮寄：登记寄出日期和配送状态
→ 系统生成最晚归还 / 最晚寄出还书日期
56. 如果出现图书缺失、地址异常、订单异常，管理员可以暂停订单并填写异常原因。
58. 15. 还书验收链路
59. 管理员输入读者编号或扫描图书编号
→ 查询未归还记录
→ 确认归还方式、归还日期 / 用户寄出日期、图书状态
→ 系统判断是否超过 1 个月期限
→ 判断图书是否损坏 / 遗失
→ 正常：更新已归还，恢复库存
→ 异常：生成逾期、损坏或押金扣罚记录
60. 这个链路结束后，会触发后续排队队列自动处理。
62. 16. 排队异常处理链路
63. 管理员进入排队管理
→ 查看图书队列、预计时间、异常记录
→ 处理资格失效、押金不足、超时未确认等记录
→ 暂停 / 取消 / 通知读者
→ 系统继续匹配下一位排队读者
64. 如果读者补缴年费或押金，管理员可以恢复排队资格。
66. 17. 逾期损坏赔偿链路
67. 系统生成逾期 / 损坏 / 遗失记录
→ 管理员进入逾期赔偿管理
→ 查询异常记录
→ 确认逾期天数、损坏情况、赔偿金额
→ 登记追责结果
→ 扣除押金或生成补缴记录
→ 更新读者信用状态
69. 18. 读者管理链路
70. 管理员进入读者管理
→ 查询目标读者
→ 新增 / 编辑 / 启用 / 停用账号
→ 维护年费状态、押金余额、寄出额度、寄回额度、信用状态
→ 保存
71. 如果读者存在未归还图书，停用账号需要二次确认。
73. 三、系统管理员业务链路
74. 19. 分类管理链路
75. 系统管理员进入分类管理
→ 新增 / 编辑分类
→ 设置父级分类和排序
→ 保存
76. 如果分类下存在图书，系统限制直接删除。
78. 20. 借阅与配送规则配置链路
79. 系统管理员进入规则管理
→ 配置最多外借 1 个月、最大借阅数量、年费标准、押金最低余额
→ 配置年度寄出次数、年度寄回次数、同城配送范围、赔偿规则
→ 保存
→ 系统将新规则用于后续借阅、排队、配送和赔偿校验
80. 这是影响所有核心业务的规则中心。
82. 21. 公告管理链路
83. 系统管理员进入公告管理
→ 新增公告
→ 填写标题、内容、展示范围
→ 发布
→ 读者首页或通知栏展示
84. 公告内容包括开放时间、借阅规则、年费说明、配送范围、寄送次数限制、新书推荐、闭馆通知等。
86. 22. 统计报表链路
87. 系统管理员进入统计报表
→ 选择时间范围
→ 系统展示馆藏、借阅、排队、配送、费用、逾期赔偿等数据
→ 管理员查看趋势分析
88. 统计结果可用于馆藏采购和运营管理。
90. 23. 日志审计链路
91. 系统记录登录日志和关键业务操作日志
→ 系统管理员进入日志管理
→ 按用户、时间、操作类型筛选
→ 查看规则调整、订单处理、配送变更、排队处理日志
→ 必要时导出
92. 日志链路用于追溯问题，尤其是订单、配送、赔偿、权限变更等关键操作。

# 云端图书馆借阅系统V2.0.docx

## Paragraphs

1. 第2章 软件需求
2. 2.1 愿景
3. 随着公共图书馆服务逐步向线上化、云端化发展，传统只能到馆检索、到馆借还的服务方式已经难以满足读者远程借阅、预约排队和配送到家的需求。图书馆需要通过统一的云端平台管理馆藏库存、读者资格、年费权益、借阅订单、同城配送、邮寄还书、排队预约和逾期处理等业务，使读者可以在线完成图书选择、借阅方式确认、地址填写和还书方式选择，管理员也能够实时掌握图书流转状态。
4. 本系统拟建设为云端图书馆借阅系统 Web 端应用，面向读者、图书管理员和系统管理员提供线上借书、到馆自取、图书馆同城邮寄、到馆还书、读者邮寄还书、排队预约、借阅记录、费用与额度查询、馆藏维护、公告通知和统计分析等功能。系统以图书流通状态为核心，结合年费、押金、配送规则和排队规则，为读者展示可借状态、当前排队人数和预计最快可借阅时间。正常情况下系统自动完成借阅校验和排队转借，管理员主要负责收发货确认、异常处理以及图书损坏追责。
5. 在技术实现上，本系统采用常见的前后端分离 Java Web 技术路线。后端采用 Java 17、Spring Boot、MyBatis-Plus、Maven、MySQL 8 和 Redis，前端采用 Vue 3、Vite、Element Plus、Pinia、Vue Router 和 Axios。系统通过云端部署提供在线访问能力，并使用数据库事务和缓存机制保障借阅库存、排队顺序、配送状态和费用额度的一致性。
6. 2.2 功能需求
7. 2.2.1 系统参与者
8. 本系统为以下角色身份提供服务：游客、读者、图书管理员、系统管理员。系统根据不同角色的业务权限，提供线上借阅、配送管理、排队管理和后台配置等功能模块。
9. 1.游客
10. 游客可以在未登录状态下浏览图书馆公告、查看公开馆藏目录、按书名或作者检索图书基础信息，并查看图书是否可借及大致排队情况。游客在进行线上借书、排队预约、查看个人权益、填写地址等需要身份识别的操作时，系统应引导其完成登录或读者注册。
11. 2.读者
12. 读者是云端借阅服务的主要对象，可以维护个人资料和同城邮寄地址，缴纳或查看年度服务费用和押金状态，检索图书，查看图书详情、可借数量、排队人数和预计最快可借时间，在线提交借阅申请，选择到馆自取或图书馆邮寄，查看当前借阅和配送状态，并在还书时选择到馆归还或邮寄到图书馆。
13. 3.图书管理员
14. 图书管理员负责图书馆日常流通业务，可以维护图书信息，处理图书入库、馆方寄出、到馆自取确认、到馆还书验收、邮寄到付收书、排队到号确认、逾期提醒、损坏追责、押金扣罚和馆藏盘点等操作。正常借阅无需管理员审批，管理员主要处理异常情况和收发货确认，确保系统状态与实际图书流转一致。
15. 4.系统管理员
16. 系统管理员负责平台基础数据和运营规则配置，可以管理用户、角色、菜单、图书分类、借阅规则、年费标准、押金标准、年度寄送次数、同城地址范围、配送费用、公告通知、统计报表和操作日志，保障系统安全、稳定和规范运行。
17. 2.2.2 功能模块描述
18. 本系统运行环境为电脑 PC 端 Web 应用。系统采用前后端分离架构，后端按用户权限、图书馆藏、线上借阅、配送流转、排队预约、费用额度、逾期赔偿、统计报表等业务模块进行组织，前端根据登录角色展示对应菜单和页面。
19. 1.用户认证与权限模块：
20. 用户可以完成登录、退出、密码修改和个人资料维护。读者个人资料应包含手机号、邮箱、常用收货地址和所在城市信息；系统应校验邮寄地址是否位于图书馆所在市内，并根据读者、图书管理员和系统管理员等角色控制菜单和接口访问范围。
21. 2.图书检索与馆藏浏览模块：
22. 读者和游客可以按书名、作者、出版社、ISBN、分类、馆藏位置等条件检索图书。系统展示图书封面、简介、分类、总馆藏、可借数量、当前排队人数、预计最快可借时间和图书流通状态。
23. 3.图书信息管理模块：
24. 图书管理员可以新增、编辑、删除和查询图书信息，维护图书分类、作者、出版社、ISBN、馆藏位置、总数量、可借数量和图书状态，并标记图书是否支持外借配送。
25. 4.借阅流通管理模块：
26. 读者可以在线提交借阅申请，并选择到馆自取或填写市内地址由图书馆邮寄。系统自动校验读者年费、押金、借阅资格、可借库存、市内地址和年度寄送次数；校验通过后自动生成借阅订单或待自取记录，不设置管理员审批环节。图书馆寄出时配送费用按现结规则处理，管理员负责寄出确认和收发货状态登记。
27. 5.预约与续借管理模块：
28. 当目标图书无剩余可借馆藏时，读者可以加入排队借阅队列。系统按照提交时间生成排队顺序，并根据每位排队读者最多外借 1 个月的规则，计算当前读者预计最快可借时间。排到队首且图书恢复可借时，系统自动校验读者资格并生成借阅订单，无异常时不需要管理员审批；若读者资格失效、押金不足或超过额度，系统跳过或暂停该排队记录并提示处理。
29. 6.逾期罚款管理模块：
30. 系统根据图书馆寄出或读者到馆自取确认日期、用户寄出还书或到馆归还日期自动判断是否超过 1 个月外借期限，并结合图书是否损坏生成逾期或赔偿记录。邮寄在途时间纳入外借期限计算，不因物流运输时间另行顺延。图书损坏、遗失或逾期未处理时，管理员可以向用户追责，并按规则从押金中扣除相应罚金或赔偿金额。
31. 7.读者管理模块：
32. 图书管理员和系统管理员可以维护读者账号、联系方式、证件信息、年度服务费状态、押金余额、年度寄出次数额度、年度寄回次数额度、借阅额度、账号状态和信用状态，便于进行借阅资格控制。
33. 8.公告通知模块：
34. 系统管理员可以发布图书馆开放时间、借阅规则调整、年费说明、配送范围、寄送次数限制、新书推荐、闭馆通知等公告。读者可在首页查看最新通知。
35. 9.统计报表模块：
36. 系统管理员可以查看图书借阅排行、馆藏数量、读者活跃度、排队预约情况、配送订单数量、年度寄送次数使用情况、押金扣罚记录、逾期赔偿记录和图书分类占比等统计信息，为馆藏采购和运营管理决策提供依据。
37. 10.系统日志与数据维护模块：
38. 系统记录登录日志、操作日志和关键业务变更记录。管理员可以查看日志、备份数据、维护借阅期限、配送范围、年费规则、寄送额度和系统参数，保证问题可追溯。
39. 2.2.3 系统业务用例

91. 2.3 非功能需求
92. 2.3.1 系统稳定性
93. 系统需要保障在多读者同时检索图书、提交线上借书、加入排队队列、查询配送状态，以及管理员处理借还书和配送订单时能够稳定运行。对于数据库连接异常、网络中断、库存扣减失败或队列计算异常，系统应提供明确提示，并尽量避免业务数据丢失。
94. 2.3.2 系统可用性
95. 系统页面应采用清晰的菜单导航和表格布局。图书检索、图书详情、借阅订单、排队队列、配送状态、费用额度和后台管理页面应提供搜索、分页、状态标签和操作提示，使读者和管理员能够快速完成常见操作。
96. 2.3.3 系统安全性
97. 系统应对登录认证、接口访问、角色权限、地址信息、年费、押金、费用记录和敏感操作进行安全控制。读者只能查看和维护自己的借阅、地址、费用和配送数据，图书管理员只能办理授权收发货和异常处理业务，系统管理员才能进行用户权限、规则配置和日志查看。
98. 2.3.4 系统可维护性
99. 系统采用前后端分离和分层设计。后端按照控制层、服务层、数据访问层和实体对象进行组织，前端按照视图、组件、接口请求和状态管理进行组织；借阅、配送、排队和费用模块保持边界清晰，便于后续维护与扩展。
100. 2.3.5 系统可扩展性
101. 系统应支持后续扩展移动端借阅查询、第三方物流接口、在线支付、短信或邮件提醒、智能推荐和多馆区调拨等功能。图书、借阅、排队、配送、费用和统计等模块应保持清晰边界。
102. 2.3.6 系统性能
103. 图书检索、分类列表、热门图书、公告信息和排队预计结果可以结合 Redis 缓存提高响应速度。图书列表、借阅记录、配送订单和费用明细应采用分页查询，避免一次加载过多数据影响体验。
104. 2.3.7 数据一致性
105. 线上借书、库存扣减、排队入列、排队出列、自动转借、馆方寄出、读者寄回、到馆归还、费用额度扣减、押金扣罚和逾期赔偿处理等流程需要保证图书库存、借阅状态、配送状态、读者权益和队列顺序一致。关键业务操作应记录状态变更时间和操作人，便于追溯。
106. 2.3.8 系统兼容性
107. 系统 Web 端应兼容主流浏览器，并适配常见电脑屏幕分辨率。前端页面组件应保持统一风格，避免在不同浏览器下出现明显布局错乱。
108. 2.4 技术与运行环境需求
109. 本系统使用 Java Web 前后端分离技术路线，支持云端部署和在线访问。系统开发、运行和测试环境需求如下。
110. 2.5 数据需求
111. 系统业务数据应围绕读者、图书、馆藏、借阅订单、排队记录、配送记录、年费权益、押金账户、地址信息和费用记录等核心对象进行建模，并通过 MySQL 进行持久化存储。主要实体说明如下。
112. 2.6 业务规则
113. 1. 未登录用户只能浏览公开图书信息，不允许提交线上借书、加入排队队列、查看个人借阅记录或维护邮寄地址。
114. 2. 读者需要具备有效年度服务资格和足额押金，且不存在逾期未处理、赔偿未处理或账号受限情况，才允许继续借阅或排队预约。
115. 3. 每位读者的最大在借数量、最多外借 1 个月期限、年度图书馆寄出次数、年度读者寄回次数、押金最低余额和排队数量由系统规则统一控制；邮寄借阅从图书馆寄出日开始计期，读者邮寄还书以用户寄出日作为期限截止判断日期。
116. 4. 读者线上借书时可选择到馆自取或图书馆邮寄；选择邮寄时必须填写图书馆所在市内地址，超出同城范围的地址不允许提交。
117. 5. 图书馆向读者寄出图书时配送费用按现结处理；读者邮寄图书归还图书馆时按到付处理；到馆自取和到馆还书不产生邮寄流程。
118. 6. 图书借出或确认寄出后，可借数量应同步减少；图书归还验收通过后，可借数量应同步恢复，并触发排队队列中下一位读者的自动资格校验和借阅订单生成。
119. 7. 当图书无可借库存时，系统允许读者加入排队队列，并根据前方排队人数及每人最晚还书日期计算预计最快可借时间；排队到号后正常情况自动进入借阅流程，异常情况由管理员介入处理。
120. 8. 管理员和图书管理员的关键操作应写入操作日志，包括规则调整、订单处理、配送状态变更、排队处理、逾期赔偿处理和账号状态变更。
